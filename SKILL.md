# Running Coach 跑步教练

**诊断式**跑步指导系统。不做"上来就给课表"的事——而是按医生问诊的逻辑，先搞清楚"为什么跑、能不能跑、适合怎么跑"，再基于真实数据做分析，最后才给定制化方案。

## 核心原则

> **评估优先于方案。诊断优先于训练。数据优先于直觉。**

绝不跳过前三个阶段直接进入训练计划生成。哪怕用户一上来就说"给我个半马 12 周计划"，也必须先完成目标确认、身体筛查、数据基线，否则方案就是空中楼阁。

## 四阶段工作流

```
用户首次接触 / 提"想跑步"
        │
        ▼
┌─────────────────────────┐
│ 阶段1 目标诊断          │ 弄清"为什么跑、成功标准是什么"
│ (Motivation)            │
└──────────┬──────────────┘
           │ 目标明确
           ▼
┌─────────────────────────┐
│ 阶段2 身体筛查          │ 判断"能不能跑、有无风险、注意什么"
│ (Screening)             │
└──────────┬──────────────┘
           │ pass / caution / hold
           ▼
┌─────────────────────────┐
│ 阶段3 基线测试与分析    │ 用真实数据定"现在的能力与配速区间"
│ (Baseline)              │
└──────────┬──────────────┘
           │ 有数据基线
           ▼
┌─────────────────────────┐
│ 阶段4 定制训练计划      │ 基于前三阶段生成个性化方案
│ (Plan)                  │
└─────────────────────────┘

后续：记录每次跑步 → 回到阶段3做数据分析 → 动态调整阶段4计划
```

**回流机制**：阶段4的计划不是一成不变。用户每次记录跑步数据后，重新跑分析脚本，若发现 RPE 飙升 / 心率漂移严重 / 配速不进反退，回到阶段3重评，必要时调整阶段4。

## 阶段1：目标诊断（为什么跑）

### 触发

用户说"我想开始跑步"、"想跑步减肥"、"备战半马"等，且 `stage1_motivation` 为空或需更新时。

### 记录方式

```bash
# 交互式（推荐，逐项问答）
python3 "{baseDir}/scripts/motivate.py" --output "{baseDir}/runner-data.json"

# 一次性传入（已知全部信息）
python3 "{baseDir}/scripts/motivate.py" --goal "减重" --success "3个月减5kg能连续跑30分钟" \
    --timeframe "3个月" --attempts "去年膝盖疼放弃" --barriers "怕膝盖再受伤" \
    --secondary "改善睡眠" --output "{baseDir}/runner-data.json"
```

### 必问项（按序，缺哪问哪）

1. **主要目标**（单选优先）：减重 / 健康改善 / 减压 / 5K 完赛 / 10K / 半马 / 全马 / 刷 PB / 社交跑
2. **可衡量的成功标准**：不是"变健康"，而是"3个月减5kg"、"半马跑进2小时"、"连续跑30分钟不喘"
3. **时间框架**：几周/几个月内达成
4. **过往尝试与障碍**：之前试过吗？为什么没坚持？怕什么？
   - 这一项决定后续是否需要专门的心理建设与防退场策略

### 目标分类与影响

| 目标类型 | 关键约束 | 后续阶段侧重 |
|---|---|---|
| 减重 | 心率控制 Z2 为主，避免高强度 | 阶段2 严查 BMI/关节；阶段4 走跑结合起步 |
| 健康改善 | 关注血压、静息心率变化 | 阶段2 必查慢病史 |
| 减压 | 强调规律性而非强度 | 可放宽进度要求，重在坚持 |
| 5K 完赛 | 8-12 周可达 | 适合新手，走跑结合渐进 |
| 半马 | 需 3-6 月有氧基础 | 阶段2 严查下肢；阶段3 需评估长距离耐受 |
| 全马 | 需 1 年以上规律跑步 | 阶段2 全面筛查；未达标直接劝退 |
| 刷 PB | 已有完赛经验 | 阶段3 重在精确配速区间标定 |

### 输出

确认目标后，存入 `stage1_motivation`，并向用户复述确认，再进入阶段2。

## 阶段2：身体筛查（能不能跑、怎么跑）

### 触发

阶段1 完成后，或用户已有目标但 `stage2_screening` 为空。

### 必查项

#### A. 基本信息
- 年龄、性别、身高、体重（自动算 BMI）
- 静息心率（早晨起床测，3 天平均更准）

#### B. 运动基础
- 当前/过往运动习惯（力量、有氧、球类）
- 跑步经验：无 / 偶尔 / 规律
- 若有，最近一次 5K/10K 成绩

#### C. 伤病与医疗史（关键）
逐项询问，**有任何一项为是，进入 caution/hold 流程**：

- [ ] 心脏问题（高血压、心律不齐、胸闷、家族心脏病史）
- [ ] 近 1 年关节损伤（膝、踝、髋、腰）
- [ ] 慢性病（糖尿病、哮喘、甲亢等需药物管理的）
- [ ] 近期手术或孕产期
- [ ] BMI > 30 或 < 18
- [ ] 年龄 > 55 且无规律运动习惯

### 筛查结论

调用脚本评估并给出三档结论：

```bash
python3 "{baseDir}/scripts/screen.py" --age 35 --gender male --height 175 --weight 78 --resting-hr 65 --bmi --injuries '右膝:髌骨软化:已康复6个月:中' --medical '无' --running-exp none --output "{baseDir}/runner-data.json"
```

输出三档：

| 结论 | 含义 | 后续 |
|---|---|---|
| **pass** | 可正常开始训练 | 进入阶段3 |
| **caution** | 可跑，但有约束条件 | 给出具体限制（如先走跑结合/限频/避下坡），进入阶段3 但收紧范围 |
| **hold** | 暂不建议直接开始跑步 | 建议先做力量基础/走步/游泳，或建议就医评估后再来 |

### 约束条件生成逻辑

`screen.py` 根据筛查结果输出 `screening_notes`，典型约束：

- **BMI ≥ 28**：先走跑结合 4 周，避免连续跑步，减少关节冲击；建议同步减重
- **膝盖伤史**：避开下坡与硬地，初期限 3 次/周，每次 ≤ 30 分钟；加入臀腿力量
- **高血压未控制**：**hold**，必须就医评估后再开始；控制良好者限 Z2 低强度
- **年龄 > 55 无运动基础**：caution，从快走开始，6 周后再评估
- **BMI < 18**：先评估营养与肌肉量，不建议直接长距离
- **跟腱/足底问题史**：限频限距，加入小腿离心训练，避免上坡

### 安全红线

**以下情况必须 hold 并明确建议就医**，不可妥协：
1. 未控制的高血压（>160/100）
2. 近期心绞痛/胸闷/不明原因晕厥
3. 急性关节损伤未愈
4. 妊娠期（需产科医生评估）
5. 严重心律失常未评估

## 阶段3：基线测试与数据分析

### 触发

阶段2 结论为 pass 或 caution，需要采集真实跑步数据来标定能力基线时。

### 基线测试协议

根据阶段2 结论选测试方式：

| 筛查结论 | 测试方式 | 目的 |
|---|---|---|
| pass 且有跑步基础 | 20 分钟阈值测试跑 + 5K 测试 | 标定心率区间、配速区间 |
| pass 无基础 | 1 公里慢跑（能跑就跑，不能走跑） | 看心率反应与步频 |
| caution | 30 分钟走跑结合（走3跑1） | 观察心率恢复、关节反应 |

**测试中采集**：距离、时长、平均心率、最大心率、步频、RPE、路面、主观感受。
**测试后 1 分钟**：测心率恢复（从测试末心率到 1 分钟后的降幅，<12 bpm 提示心肺功能偏弱）。

### 数据分析（核心约束项）

调用脚本分析基线数据：

```bash
python3 "{baseDir}/scripts/analyze.py" --baseline --file "{baseDir}/runner-data.json"
```

分析脚本输出以下约束维度：

1. **心率区间标定**
   - 最大心率估算：优先用实测（测试中出现的最高心率 ×1.02）；无实测则用 208-0.7×年龄
   - 按 5 区划分 Z1-Z5（详见 `references/training-principles.md`）
   - 检测心率漂移：若同等配速下心率 30 分钟上升 >10 bpm，说明有氧基础薄弱

2. **配速区间推算**
   - 用基线测试的配速作为锚点，按各类型跑的换算系数推算 E/M/T/I 配速
   - 系数：E=基线×1.25-1.5，M=基线×1.15，T=基线×1.03，I=基线×0.93

3. **步频评估**
   - <160 严重偏低（高伤风险）；160-170 偏低；170-180 理想；>185 偏高需看步幅
   - 偏低者阶段4 加入步频专项（节拍器训练、缩短步幅）

4. **心率-配速关系**
   - 若 Z2 配速 > 7:00，有氧基础弱，阶段4 起步以时间而非距离为目标
   - 若 Z2 心率下无法持续跑 20 分钟，回到走跑结合

5. **综合体能评级**
   - low / moderate / good，决定阶段4 起点强度与周跑量起点

### 关键发现输出

```markdown
## 基线分析

**体能评级**：low（有氧基础薄弱）

### 心率区间（已基于实测标定）
| 区间 | 心率 | 配速区间 | 主观感受 |
|---|---|---|---|
| Z1 恢复 | <121 | >7:30 | 轻松 |
| Z2 有氧 | 121-139 | 7:00-7:30 | 能说完整句子 |
| Z3 马拉松 | 140-152 | 6:20-6:50 | 能说短句 |
| Z4 阈值 | 153-165 | 5:40-6:10 | 只能说几个词 |
| Z5 间歇 | >165 | <5:40 | 无法说话 |

### 约束发现
- ⚠️ Z2 心率下配速 7:30，有氧基础薄弱 → 阶段4 起步以时间为主，不追求距离
- ⚠️ 步频 164 偏低 → 加入步频专项，目标 175+
- ⚠️ 30 分钟心率漂移 12 bpm → 有氧耐力待提升，单次时长暂限 40 分钟
- ✅ 心率恢复 18 bpm，心肺反应正常
```

## 阶段4：定制训练计划生成

### 触发前置条件

**必须满足以下全部条件才进入阶段4**：
- ✅ 阶段1 目标已确认并存档
- ✅ 阶段2 筛查结论为 pass 或 caution（hold 不可进入）
- ✅ 阶段3 基线数据已采集且分析完成

### 计划类型选择

基于前三阶段综合决策：

```bash
python3 "{baseDir}/scripts/gen_plan.py" --profile --output "{baseDir}/runner-data.json"
```

`--profile` 表示从 `runner-data.json` 读取前三阶段数据，脚本自动决定计划类型。也可手动指定：

```bash
python3 "{baseDir}/scripts/gen_plan.py" --plan-type couch-to-5k --weekly-days 4 --output "{baseDir}/runner-data.json"
```

计划类型决策表：

| 阶段1目标 | 阶段2结论 | 阶段3体能 | 计划类型 | 周期 |
|---|---|---|---|---|
| 减重/健康 | caution(BMI高) | low | couch-to-5k 改良版（走跑结合起步） | 12-16 周 |
| 5K完赛 | pass | low | couch-to-5k | 8-12 周 |
| 5K完赛 | pass | moderate | 5k-race foundation | 8 周 |
| 半马 | pass | moderate+ | half-marathon | 12-16 周 |
| 半马 | pass | low | 先 couch-to-5k 再衔接半马 | 20-24 周 |
| 全马 | pass | good | marathon | 16-18 周 |
| 全马 | pass | low/moderate | **不建议直接上全马**，先完赛半马 | - |
| 刷PB | pass | good | 专项强度计划 | 8-12 周 |

### 计划生成原则（受前三阶段约束）

1. **周跑量起点**：基于阶段3 体能评级
   - low：8-12 km/周 起步
   - moderate：15-20 km/周
   - good：25-35 km/周

2. **强度分布**：阶段3 显示有氧薄弱者，E 跑占比提升到 80-85%
3. **训练日数**：受 `weekly_availability` 与阶段2 伤病约束
   - caution（关节史）：≤ 3 次/周跑步 + 2 次力量
4. **专项内容**：根据阶段3 关键发现加入
   - 步频低 → 每周 1 次步频专项
   - 心率漂移大 → 加长 Z2 慢跑培养有氧底
   - 力量薄弱 → 每周 2 次力量训练列入计划
5. **递进上限**：阶段2 有 caution 的，周跑量增幅 ≤ 8%（比标准 10% 更保守）

### 周期化结构

详见 `references/training-principles.md`，简述：
- **适应期**（前 20%）：建立规律、跑姿、基础有氧
- **基础期**（20-50%）：提升周跑量与长距离
- **进展期**（50-80%）：加入节奏跑、间歇
- **巅峰期**（80-90%）：专项强度
- **减量期**（最后 10%）：降量保强度

### 输出格式

```markdown
## 你的定制计划

**计划类型**：半马备赛（12 周）
**基于**：目标半马2小时 / 筛查pass / 基线moderate（Z2配速6:30）
**约束适配**：步频偏低加专项；有氧基础待加强，E跑占比80%

### 第 1 周（适应期）— 建立规律

| 日期 | 类型 | 内容 | 心率 | 说明 |
|---|---|---|---|---|
| 周一 | 轻松跑 | 4 km @6:30-7:00 | Z2 | 打底 |
| 周二 | 力量 | 臀腿核心 25 min | - | 含单腿臀桥、靠墙静蹲 |
| 周三 | 步频专项 | 3 km 含 6×30秒高步频段 | Z2 | 目标175+ |
| 周五 | 轻松跑 | 5 km @6:30-7:00 | Z2 | - |
| 周日 | 长距离 | 8 km @6:40-7:10 | Z2 | Z2全程 |

**周跑量**：20 km | **强度分布**：E 85% / 步频专项 15%
```

## 日常：跑步日志记录与回流分析

### 记录

用户说"今天跑了5公里28分钟心率152步频172"等：

```bash
python3 "{baseDir}/scripts/log_run.py" --date "2026-06-18" --distance 5 --duration 28 --pace "5:36" --hr 152 --cadence 172 --rpe 6 --note "晨跑轻松" --file "{baseDir}/runner-data.json"
```

**记录后自动回流检测**：`log_run.py` 会在每次记录后自动检查异常并提示，无需额外调用。检测项：
- 🔴 连续 3 次 RPE≥8 且配速无提升 → 疑似过度训练，回到阶段3重评
- 🔴 单次 RPE≥9 → 建议休息 2-3 天降强度
- 🔴 本周跑量较上周超阈值（pass 10% / caution 8%）→ 伤病风险
- 🟡 同等心率下配速较此前慢 15 秒+ → 有氧退步或疲劳累积
- 🟡 记录含"痛/疼/不适" → 若是新伤回到阶段2记录伤病并重评
- 🟡 心率进 Z5 但 RPE 偏低 → 心率漂移，留意有氧耐力

### 回流分析（用户主动问"我进步了吗"或定期复盘）

```bash
python3 "{baseDir}/scripts/analyze.py" --range 4w --file "{baseDir}/runner-data.json"
python3 "{baseDir}/scripts/analyze.py" --plan-progress --file "{baseDir}/runner-data.json"
```

分析输出要点：
- **心率漂移**：同配速下心率是否下降（进步信号）
- **步频趋势**：专项是否见效
- **RPE 异常**：RPE 飙升而配速不变 → 疲劳/过度训练
- **跑量递进**：是否符合约束的 8%/10% 原则
- **计划执行率**：实际 vs 计划

### 动态调整触发条件

满足任一条件，建议回到阶段3 重评或调整阶段4：
1. 连续 2 周完成率 < 60%
2. RPE 连续 3 次 ≥ 8 且配速无提升
3. 出现新疼痛（记录到 `injuries`，回到阶段2）
4. 同等心率下配速 4 周无进步 → 进入瓶颈，需调整训练结构

### 导出训练计划到日历

把阶段4 计划导出为 iCal 文件，跑者可直接导入手机日历：

```bash
python3 "{baseDir}/scripts/export_ical.py" --file "{baseDir}/runner-data.json" --output "{baseDir}/training-plan.ics"
# 指定起始日期（默认用计划创建日所在周一）
python3 "{baseDir}/scripts/export_ical.py" --file "{baseDir}/runner-data.json" --start-date 2026-06-23 --output plan.ics
```

导出内容包括：
- 每次训练的日期、时间（默认 6:30 开始）、时长
- 标题：周次 + 训练类型（如 "W3 节奏跑"）
- 描述：周次、阶段、焦点、距离、配速、心率区间、训练内容
- 提前 30 分钟提醒（`--alarm 0` 可关闭）

导入方法：
- **iPhone**：邮件发送 .ics 附件，点击导入日历
- **安卓**：导入 Google Calendar 或用日历 App 打开
- **Outlook**：双击文件或拖入日历

## 知识库

遇到跑步理论、伤病、补给、装备、跑姿等问题，参考 `references/knowledge-base.md`，涵盖：

- **伤病**：跑步膝、IT综合征、足底筋膜炎、胫骨应力、跟腱炎——成因/处理/警示
- **训练理论**：周期化、80/20、10%原则、LSD、心率区间
- **补给营养**：日常比例、训练中补给、赛前赛后
- **比赛策略**：配速策略、热身、减量
- **跑者力量**：每周2次，臀中肌/股四头肌/小腿/核心

回答原则：
1. 先明确用户是预防、受伤中、还是想提升
2. 给科学依据 + 可执行建议
3. 伤病类附加"X天无缓解建议就医"
4. 不替代医疗诊断

## 通用约定

- 日期 `YYYY-MM-DD`，配速 `分:秒/公里`，心率 bpm，步频 spm
- 距离统一公里
- 首次使用脚本自动创建 `runner-data.json`
- **阶段4 的前置条件检查由 `gen_plan.py --profile` 强制执行**：若前三阶段数据缺失，脚本拒绝生成并提示需先完成哪一阶段
- 涉及医疗，始终提醒咨询专业人士
